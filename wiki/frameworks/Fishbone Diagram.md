# Fishbone Diagram

## Definition

The Fishbone diagram (also called the Ishikawa diagram or cause-and-effect diagram) is a graphical root-cause analysis tool: the problem sits at the "head" of the fish, and the major cause categories branch off the central "spine" as "bones," with sub-causes branching from each bone. Created by Japanese quality-control expert **Dr. Kaoru Ishikawa** to help workers avoid fixing symptoms instead of root causes, it became one of the seven fundamental quality tools and is the canonical "analyze" phase tool in **Six Sigma DMAIC** (Define, Measure, Analyze, Improve, Control). Pereira positions it as a brainstorming technique because the cause-generation phase is itself a structured group ideation exercise — the diagram is both the prompt and the output. Standard cause categories include People, Process, Equipment, Materials, Environment, and Methods (the "6 Ms" in manufacturing) — but categories adapt to context (the 8Ps in marketing: Product, Price, Place, Promotion, Personnel, Process, Physical evidence, Performance).

## Key attributes

- **Visual structure** — head (problem) + spine + bones (cause categories) + sub-bones (specific causes)
- **Category-driven** — pre-set categories prompt comprehensive coverage
- **Group brainstorming embedded** — the cause-generation phase is a team exercise
- **Iterative depth** — keep asking "why does this happen?" to add sub-bones
- **Standard category sets** — 6Ms (manufacturing), 8Ps (marketing/service)
- **One of seven basic quality tools** — Ishikawa's enduring contribution
- **DMAIC fit** — slots into Six Sigma's Analyze phase

## When to use

- Investigating a recurring problem or quality issue
- Six Sigma / Lean DMAIC analyze phase
- Manufacturing defect investigation
- Service-quality complaints with multiple plausible causes
- Customer-churn root-cause analysis (the software-subscription example below)
- Process bottleneck investigation
- Cross-functional problem-solving where each function may own a cause

## When NOT to use

- When the cause is already known and obvious
- For pure ideation (use Classic Brainstorming or SCAMPER)
- For decision-making between alternatives (use Six Thinking Hats)
- When the team lacks domain expertise to populate the bones
- For exploring future possibilities (use Futures Wheel)

## Common pitfalls

- Listing symptoms as causes (defeats the point — Ishikawa created the tool against this)
- Stopping at the first level of bones (the sub-causes are where the insight lives)
- Cramming too many causes per bone (diagram becomes unreadable)
- Picking categories that don't fit the problem (force-fitting 6Ms to a marketing issue)
- Skipping team diversity — homogeneous team produces homogeneous causes
- Treating the diagram as the deliverable rather than as input to fix decisions

## Key metrics

- **Causes identified per category** (target balance across all categories)
- **Sub-cause depth** (deeper = more thorough)
- **Causes addressed** (% of identified causes that got fix actions)
- **Recurrence rate** (did the problem return after the fix? — the real measure)

## Decision questions

- What is the precisely stated problem? (the "head" must be specific)
- Which category set fits — 6Ms, 8Ps, or custom?
- Have we asked "why?" enough times per cause to reach the root?
- Which causes are highest leverage to fix?
- Who validates the proposed root causes with data, not just brainstorming?

## Canonical examples

- **Supermarket chain delivery issues** — items mispicked/delayed/damaged; categories: Materials, Measurements, Personnel, Machines, Environment (page 113)
- **Software-subscription cancellation** — 55% churn after month one; categories included payment process, marketing, lack of incentives, weak UX (page 114)
- **Manufacturing defect investigation** — the canonical Ishikawa use case
- **Healthcare adverse-event analysis** — root-cause investigation in clinical settings (page 109)
- **Service-industry complaint analysis** — applied broadly across hospitality, retail, IT
