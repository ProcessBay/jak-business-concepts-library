# IT Outsourcing (ITO)

## Definition

IT Outsourcing (ITO) is the use of external service providers to deliver IT-based solutions and infrastructure that support or optimize parts of the business process. Pereira frames it as a flagship success story for outsourcing in general: information technology was the first function for which strategic outsourcing was recognized (Kodak, 1989), and it remains one of the most outsourced functions worldwide — 64% of companies outsource software development. The original driver was scarce expertise; today, the driver mix has expanded to cost, time, efficiency, currency with a fast-moving field, and access to security expertise that few firms can build alone. ITO spans cloud computing, SaaS, cybersecurity, data science, computer networking, software development, and helpdesk.

## Key attributes

- **High-specialization, fast-moving field** — knowledge depreciates quickly; vendors stay current at scale (page 32)
- **Spans the full IT stack** — cloud, SaaS, cybersecurity, data science, networking, software dev (page 32)
- **Top-three outsourced function** — alongside accounting and digital marketing (page 17)
- **64% of companies outsource software development** (page 17)
- **Periodic, project-shaped demand** — many IT needs are not steady-state, favoring on-demand contracting (page 33)
- **Cost, time, and efficiency advantage** — well-documented vs in-house equivalents (page 32)
- **Cybersecurity expertise as a primary motive** — modern ITO is increasingly about access to security specialists (page 33)

## When to use

- Functions where the in-house team cannot keep up with the field's pace of change
- Project-shaped IT work (implementation, migration, upgrade) with finite duration
- When in-house cybersecurity capability is below the threat profile
- When cloud or SaaS providers can deliver capability faster than internal build
- When IT is operationally essential but not core to the firm's value proposition
- When demand is variable and FTE utilization would be poor

## When NOT to use

- When IT *is* the product (most software companies keep core engineering in-house)
- For systems holding the firm's primary IP or competitive advantage
- When data sovereignty or regulatory constraints prohibit vendor access
- When the firm cannot evaluate vendor security posture
- When vendor dependency would compromise resilience (single point of failure)

## Common pitfalls

- **Handing over delicate or confidential information** without adequate controls (page 33)
- **Outsourcing core operations** whose failure would be existential (page 33)
- **Cybersecurity blind spots** — assuming the vendor's security is adequate without auditing it
- **Treating ITO as pure cost reduction** and ignoring strategic, capability, and resilience dimensions
- **Vendor lock-in via proprietary integrations** that make exit prohibitive
- **Skill atrophy in-house** — losing the ability to evaluate or replace the vendor

## Key metrics

- **Unit cost per seat / per workload** vs in-house baseline
- **Uptime / availability** against SLA
- **MTTR** (mean time to recovery)
- **Security incident rate** and time-to-patch
- **Technology currency lag** — how far behind cutting-edge the vendor keeps you
- **Contractor vs FTE cost delta** including governance overhead

## Decision questions

- Is this IT capability core to our product, or operational support?
- Can we evaluate the vendor's security posture, or are we trusting blindly?
- What happens to operations if this vendor fails or exits the relationship?
- Are we keeping enough in-house IT literacy to govern the vendor effectively?
- What's the total cost — vendor fees plus integration plus governance plus exit?

## Canonical examples

- **Eastman Kodak (1989)** — outsourced its IT systems; widely cited as the first recognized strategic outsourcing event (page 16)
- **Cloud infrastructure (AWS, Azure, GCP)** — IT infrastructure as managed service
- **SaaS adoption** — Salesforce, Workday, ServiceNow replacing in-house systems
- **Managed Security Services (MSSP)** — 24/7 SOC as a service
- **Offshore software development centers** — common in India, Eastern Europe, Latin America
- **64% software-development outsourcing rate** — including a large number of medium and large enterprises (page 17)
