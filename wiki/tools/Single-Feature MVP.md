# Single-Feature MVP

## Definition

A Single-Feature MVP is a high-fidelity MVP that contains only the **core feature** of your future product — nothing else. The hypothesis is that the one feature, on its own, is valuable enough that early users will adopt, use, and (ideally) pay for it. Pereira positions it as a high-fidelity alternative to landing-page or smoke-test MVPs: you actually build the thing, but you build only the one piece that proves the core value proposition. This forces the team to confront the hardest question — "is our value prop real?" — before getting distracted by adjacent features.

## Key attributes

- **One feature only** — the core that delivers the value proposition
- **Real, working software** — not a prototype or mockup
- **High-fidelity** — users get a complete experience of that one feature
- **Forces feature discipline** — every "and also..." is deferred to post-validation
- **Easier to test than feature-rich MVPs** — users focus on the main purpose
- **Faster to ship than full-product builds** — but more work than landing-page MVPs
- **Marketing-amplifiable** — a single sharp claim is easier to communicate

## When to use

- When you already know the market exists but want to validate whether your specific solution resonates (page 38)
- For products where the value prop hinges on one capability working really well
- Mobile and SaaS contexts where users abandon multi-feature products that do nothing well
- When you want a foothold in a market by addressing one specific need exceptionally
- After concierge or landing-page validation confirms demand and you need to test the actual product

## When NOT to use

- When the value proposition genuinely requires feature interaction (some marketplaces, some platforms)
- When competitors already offer the single feature for free
- When you don't yet know which feature is the core (do concierge or landing-page first)
- Hardware or regulated products with long build cycles
- When the "single feature" actually requires significant infrastructure to deliver

## Common pitfalls

- **Scope creep** — adding "just one more feature" because the team is uncomfortable with how minimal it looks
- **Building the wrong single feature** — picking based on team intuition, not customer evidence
- **Treating the single-feature MVP as the product** — never iterating after launch
- **Marketing the future product, not the actual MVP** — sets users up for disappointment
- **Skipping retention measurement** — initial signups are not validation; repeat usage is
- **Ignoring users who churn after the first session** — the most valuable feedback

## Key metrics

- Activation rate (% reaching the value moment)
- Retention (D1, D7, D30 cohort retention)
- Conversion (% who take the desired action: signup, purchase, share)
- Feature engagement (sessions per user, time per session on the core feature)
- NPS for the single feature ("would you recommend X to a friend?")

## Decision questions

- What single feature, in isolation, delivers our core value proposition?
- If users use only this feature and nothing else, are they better off?
- Can we ship it in weeks, not months?
- What signal would tell us the feature itself — not our marketing — is what's working?
- What's our trigger to expand from one feature to two?

## Canonical examples

- **Foursquare** — launched in 2009 with one feature: location check-ins that awarded virtual points. Added recommendations and city guides only after validation (page 79)
- **Twitter** — released as 140-character public messaging only; everything else (DMs, media, retweets, threads) came later (page 80)
- **Instagram** — launched October 2010 with the single combined feature of "take a photo, apply a filter, share it." Hit 24,000 users in 24 hours (page 107)
- **Facebook (early)** — text posts on virtual "boards" for college students; 5M users by late 2005 before any other feature mattered (page 79)
- **Amazon (early)** — affordable books online, period. Single-category, easy-to-use, validated before expanding to everything (page 81)
- **WhatsApp (early)** — status messages, then 1:1 text only
